2. Requisitos do Laravel

Laravel exige uma base técnica clara para funcionar bem no treinamento e no desenvolvimento profissional. O aluno precisa entender esse ambiente antes de criar a primeira aplicação.

Ambiente base

⚙️ Requisitos principais

PHP e Composer

São a base para instalar o framework e gerenciar dependências. Sem eles, o projeto Laravel não nasce corretamente.

Banco de dados

SQLite, MySQL e PostgreSQL aparecem muito. Mesmo em aula, isso precisa ser explicado porque migrations e models dependem dessa camada.

🧰 Ferramentas de apoio

Terminal e Artisan

Criação de projeto, servidor local, migrations, geração de controllers e limpeza de cache passam por comandos Artisan.

Node.js e npm

Entram quando há Vite, assets compilados, Alpine.js, Inertia ou outras camadas de frontend moderno.

📋 Resumo técnico

PHP
Composer
Banco de dados
Terminal / Artisan
Node.js e npm quando houver frontend compilado

Esse resumo deixa claro ao aluno que Laravel pode ser simples no início, mas cresce com ferramentas adicionais conforme o kit escolhido.

3. Onde e como o Laravel surgiu

Laravel surgiu em 2011, foi criado por Taylor Otwell e tem origem nos Estados Unidos. A proposta era tornar o desenvolvimento em PHP mais elegante, produtivo e organizado, com foco na experiência de uso do framework.

Origem do framework

🗓️ Linha do tempo

Ano de criação

2011 marca o início público do Laravel como framework moderno para aplicações web em PHP.

Criador

Taylor Otwell é o principal nome associado à criação, direção e ao amadurecimento do ecossistema Laravel.

País de origem

O framework nasceu nos Estados Unidos, dentro do contexto de evolução do ecossistema PHP moderno.

🏢 Empresa e ecossistema

Ecossistema oficial

Com o crescimento do framework, a marca Laravel passou a reunir serviços, produtos e ferramentas oficiais ao redor da plataforma.

Motivação técnica

A proposta era entregar uma sintaxe mais elegante e um conjunto mais completo de recursos prontos para rotas, views, banco, validação e autenticação.

Evolução de plataforma

Com o tempo, Laravel deixou de ser visto apenas como framework e passou a ser percebido como um ecossistema completo de desenvolvimento.

🎓 Leitura para o treinamento

Por que isso importa

Ao aprender a origem do Laravel, o aluno entende que ele nasceu para melhorar a experiência de desenvolvimento e não apenas para adicionar bibliotecas soltas a um projeto PHP.

Mensagem pedagógica

Esse contexto ajuda a mostrar ao aluno que Laravel foi desenhado para produtividade, legibilidade e organização desde a origem.

Tom premium do material

A seção agora fica equilibrada visualmente, com três blocos bem resolvidos e a mesma densidade de leitura em cada coluna.

4. Blade

Blade é o caminho mais direto para entregar interface em Laravel. Ele já faz parte do ecossistema do framework e serve muito bem quando a aplicação já está funcionando em renderização no servidor.

Já vem com Laravel

Instalação e estado atual

Já vem instalado

Blade faz parte do Laravel. Em projeto novo ou já existente, você normalmente já pode criar views em `resources/views` sem instalar nada extra.

Entrada técnica

Usa rotas, controllers, componentes Blade, layouts e includes. A renderização acontece no servidor e o HTML chega pronto ao navegador.

Quando usar

Melhor para

CRUD tradicional, painéis internos, páginas institucionais, autenticação e sistemas que não exigem interações muito ricas no frontend.

Complexidade

Baixa a média. É a opção mais segura para começar e a mais fácil de manter quando o time é mais backend.

Migração em projeto existente

Se já está em Blade

Você não precisa trocar nada. O caminho técnico normalmente é organizar melhor layouts, partials e componentes para ganhar produtividade.

Estrutura funcional

`routes/web.php`, controllers, views Blade, formulários com validação, sessão, middleware e Eloquent trabalham juntos sem camada extra.

5. Blade + Alpine.js

Alpine.js entra quando a aplicação Blade já roda bem, mas você precisa de comportamento no navegador sem subir para uma SPA. Ele resolve estado pequeno e interação leve no próprio HTML.

Precisa adicionar Alpine

O que vem e o que instala

Blade já existe

Se o projeto já está em Blade, a base visual e de rotas já está pronta.

Alpine precisa entrar

Geralmente via Vite e npm. Depois você usa diretivas como `x-data`, `x-show`, `x-on` e `x-model` nos templates.

Quando faz sentido

Interações locais

Dropdowns, abas, modal, accordion, busca simples, máscara visual, alternância de estado e pequenos comportamentos de UI.

Não é ideal para

Tela muito rica em estado compartilhado, dashboard muito dinâmico ou produto com fluxo parecido com SPA completa.

Projeto Blade já rodando

Adoção gradual

Você pode iniciar em uma tela específica sem reescrever tudo. Alpine é excelente para evolução incremental.

Fluxo técnico

Instalar dependência, registrar no bundle do Vite, compilar assets e aplicar as diretivas apenas onde a interação for necessária.

6. Blade + Livewire

Livewire e uma evolucao natural para quem esta em Blade e quer mais reatividade sem sair do ecossistema Laravel. O componente mora no servidor, mas a experiencia do usuario fica mais dinamica.

Precisa instalar Livewire

Base tecnica

Continua em Laravel

Rotas, validação, policies, middleware, sessao e models continuam no mesmo fluxo do framework.

Componente reativo

A UI conversa com o backend por atualizacoes parciais. E muito util para tabelas, filtros, wizard, formularios e blocos com refresh seletivo.

Ja tenho Blade funcionando

Migracao tranquila

Você pode escolher apenas um formulario ou uma listagem para virar componente Livewire. Não precisa refazer o projeto inteiro.

Melhor encaixe

Backoffice, CRM, ERP, painéis administrativos, tela de cadastro com feedback dinamico e acoes em massa.

Ponto funcional

Menos separacao de camadas

O time continua pensando em Laravel end to end. Isso reduz atrito quando nao ha uma equipe frontend dedicada.

Atencao

Se a interface exigir estado muito complexo no cliente, animacoes ricas e navegação de SPA, talvez Inertia faca mais sentido.

7. React + Inertia

Inertia merece atenção especial porque ele muda a estratégia de integração. Em vez de separar logo de início backend e frontend em duas aplicações, o Laravel continua respondendo rotas e controllers, enquanto o React assume a pagina.

Inertia e a ponte

O que e Inertia

Não é API primeiro

Você mantem Laravel como controlador do fluxo. Controllers retornam páginas Inertia, não apenas JSON para um frontend separado.

Troca de páginas com cara de SPA

A navegação fica fluida, mas autenticação, middleware, validação e autorização continuam confortáveis dentro do Laravel.

Instalação e adoção

Precisa instalar

Inertia e React não chegam prontos em um projeto Blade tradicional. E preciso adicionar dependencias, configurar o adapter e criar as páginas React.

Projeto Blade já rodando

Você pode iniciar por um módulo específico, como dashboard ou área logada. Não precisa migrar todo o sistema de uma vez.

Quando usar

Bom para

Produtos com UX mais rica, dashboards, navegação intensa entre páginas, estado mais complexo e equipe acostumada com React.

Evita separar cedo demais

Você ganha modernização de frontend sem cair logo na manutenção de API, CORS, auth desacoplada e dois deploys independentes.

8. Vue + Inertia

Vue + Inertia segue a mesma arquitetura funcional do React + Inertia. O que muda não é a ideia central, e a biblioteca de interface escolhida para construir as páginas do lado do cliente.

Mesmo fluxo do Inertia

O que já existe

Laravel continua central

Rotas, middleware, requests, regras e persistência continuam no backend Laravel normalmente.

Vue e Inertia entram depois

A instalação exige adapter Inertia, dependência do Vue e configuração no pipeline de build com Vite.

Projeto em Blade

Entrada por módulos

Uma pratica funcional e manter parte em Blade e mover apenas areas com maior dinamismo para páginas Inertia com Vue.

Força do Vue

Boa ergonomia para componentes, formulários ricos, composição visual e curva amigável para equipes mistas.

Fluxo técnico

composer create-project laravel/laravel nome-do-projeto
php artisan serve
php artisan migrate
npm install
npm run dev

Depois entram as dependencias do Inertia e do Vue, as páginas em `resources/js/Pages` e os controllers retornando respostas Inertia.

4. Criando um projeto Laravel

Depois de entender conceito, requisitos e origem, o fluxo correto do treinamento e criar a aplicacao. Aqui o aluno precisa ver não apenas os comandos, mas a logica de preparacao, execucao e primeiros passos logo depois da instalacao.

Primeiro projeto

🧭 Antes de começar

Verifique PHP, Composer e banco. Se houver kit frontend moderno, prepare também Node.js e npm.

⚙️ Execução mínima

Criar projeto, entrar na pasta, subir servidor e aplicar migrations já entrega uma base funcional para aula e testes.

🚀 O que vem depois

Depois da aplicação rodar, o aluno está pronto para entender a estrutura de pastas e só então decidir o kit frontend.

🚀 Fluxo inicial

composer create-project laravel/laravel treinamento-laravel
cd treinamento-laravel
php artisan serve
php artisan migrate
Sequência mínima

Esse é o caminho clássico para começar com uma aplicação Laravel rodando localmente sem desviar do essencial.

Bom para treinamento

A ordem é simples, visual e fácil de explicar em aula, especialmente para quem está vendo Laravel pela primeira vez.

Tempo de preparação

Esse fluxo mostra bem em aula que um projeto Laravel pode sair do zero para uma base executando em poucos minutos quando o ambiente já está pronto.

🧩 O que acontece em cada comando

Criação do projeto

O Composer baixa o esqueleto do framework, instala dependências PHP e monta a estrutura inicial da aplicação.

Entrada na pasta

Ao entrar na pasta do projeto, todos os comandos seguintes passam a agir sobre a aplicação correta.

Servidor local

`php artisan serve` sobe o ambiente de desenvolvimento para o navegador e facilita a demonstração no treinamento.

Migrations

`php artisan migrate` aplica a estrutura inicial do banco e introduz persistência de forma prática no material.

📦 Próximos passos e cenários

Quando entra frontend compilado

Se o projeto usar Vite, Alpine.js, Inertia ou qualquer kit com assets modernos, entram também os comandos abaixo.

npm install
npm run dev
Leitura didática correta

Depois de criar e rodar o projeto, o próximo passo do treinamento deve ser explicar a estrutura de pastas antes de pular para os kits.

Ponto de transição

Essa etapa fecha o bloco de inicialização e abre naturalmente a leitura da estrutura interna do framework.

Próxima leitura natural

Com o projeto no ar, a seção seguinte deve mostrar pastas, responsabilidades e fluxo interno do Laravel com calma.

11. Svelte + Inertia

Svelte + Inertia reaproveita a mesma base de integração do Inertia, mas troca React ou Vue por uma abordagem mais enxuta no frontend. Isso pode agradar times que querem menos cerimônia no código da interface.

Inertia com frontend enxuto

Instalação e encaixe

Precisa instalar

Assim como nas outras variações com Inertia, um projeto Blade puro não traz isso pronto. Será preciso incluir Inertia, Svelte e a configuração de build.

Pode coexistir com Blade

Nada impede manter páginas antigas em Blade e criar novas telas com Inertia + Svelte apenas onde fizer sentido.

Funcionalmente forte em

Interfaces rápidas

Boa opção para experiências fluidas, menos boilerplate e componentes enxutos com boa percepção de performance.

Ponto de decisão

A decisão principal não é o Svelte em si, e sim escolher se a estratégia Inertia faz sentido para o produto e para o momento da aplicação.

Fluxo funcional

composer create-project laravel/laravel treinamento-laravel
cd treinamento-laravel
php artisan serve
php artisan migrate
npm install
npm run dev

Com isso, o projeto fica pronto para receber as páginas em JavaScript, enquanto o Laravel continua sendo o eixo de autenticação, regras e navegação do servidor.

12. Filament Especial

Filament fica dentro do agrupamento de kits de frontend porque ele é uma decisão de interface e produtividade sobre o ecossistema Laravel, especialmente em painéis administrativos.

Kit admin premium

✨ O que ele entrega

Painel administrativo pronto

Dashboards, formulários, tabelas, filtros, ações e navegação administrativa com alto ganho de produtividade.

CRUD rico

Muito forte para backoffice, ERP, CRM, gestão interna e módulos administrativos com menos código manual.

🧩 Como se encaixa

Precisa instalar

Filament não vem por padrão em um Laravel novo. Ele entra como biblioteca especializada em interface administrativa.

Convive com o projeto

Pode coexistir com Blade, Livewire e outras partes do sistema quando a área admin precisa ganhar velocidade.

🎓 Papel didático

No treinamento, ele aparece depois da base, da criação do projeto e da estrutura, porque é uma camada de produtividade e não o ponto inicial para explicar o framework.

🧠 app/

Esse diretório concentra a maior parte da lógica principal da aplicação. É onde o aluno começa a entender classes de domínio, regras de negócio e componentes centrais do projeto.

📌 O que normalmente aparece aqui

  • Http/Controllers/*.php: controllers que recebem a requisição e coordenam a resposta.
  • Models/*.php: models Eloquent ligados às tabelas do banco.
  • Providers/*.php: service providers usados para registrar serviços e configurações do framework.
  • Jobs/*.php: tarefas para filas e processamento assíncrono.
  • Events/*.php e Listeners/*.php: eventos e ouvintes para desacoplar ações.
  • Console/Commands/*.php: comandos Artisan personalizados.

🎓 Leitura de treinamento

Quando o aluno quer descobrir onde mora a regra do sistema, normalmente ele acaba chegando em app/. Essa é a pasta mais importante para explicar arquitetura e responsabilidade no Laravel.

🚀 bootstrap/

Essa pasta participa da inicialização do framework. Ela ajuda a preparar o ambiente para que a aplicação Laravel suba de forma organizada.

📌 Arquivos e itens comuns

  • app.php: arquivo central de bootstrap da aplicação.
  • cache/*.php: arquivos de cache gerados pelo framework, como cache de configuração e rotas.

🎓 Leitura de treinamento

O aluno não costuma editar muito essa pasta no início, mas precisa saber que ela existe porque faz parte da partida interna do Laravel.

⚙️ config/

Aqui ficam os arquivos de configuração da aplicação. Essa pasta conversa diretamente com variáveis do .env e define comportamentos globais.

📌 Arquivos mais comuns

  • app.php: nome da aplicação, timezone, locale e configurações gerais.
  • database.php: conexões e opções do banco de dados.
  • cache.php: driver e comportamento do cache.
  • queue.php: filas e conexões assíncronas.
  • mail.php: envio de e-mails.
  • filesystems.php: discos locais, públicos e cloud.
  • services.php: integrações externas e credenciais de serviços.
  • auth.php: autenticação, guards e providers.

🎓 Leitura de treinamento

Sempre que o aluno perguntar “onde eu configuro isso?”, existe uma boa chance de a resposta estar em config/.

🗄️ database/

Essa pasta concentra a estrutura evolutiva do banco. Ela é essencial para explicar versionamento de schema, dados iniciais e geração de massa de teste.

📌 Pastas e arquivos comuns

  • migrations/*.php: arquivos de migration para criar e alterar tabelas.
  • seeders/*.php: seeders para popular dados iniciais.
  • factories/*.php: factories usadas em testes e geração de dados.
  • database.sqlite: arquivo SQLite quando esse banco é usado no projeto.

🎓 Leitura de treinamento

Essa pasta ajuda o aluno a entender que o banco também faz parte do versionamento do projeto, e não deve ser tratado como algo separado da aplicação.

🌐 public/

É a porta de entrada pública da aplicação. Tudo que o navegador pode acessar diretamente passa por essa pasta.

📌 Arquivos e itens comuns

  • index.php: arquivo principal de entrada da aplicação.
  • favicon.ico: ícone do site.
  • robots.txt: instruções para robôs e indexadores.
  • build/*: assets compilados pelo Vite.
  • storage: link simbólico para arquivos públicos quando usado php artisan storage:link.

🎓 Leitura de treinamento

O aluno precisa entender que essa é a camada exposta ao navegador. Nem tudo no projeto deve ficar acessível diretamente, e por isso essa pasta é estratégica.

🎨 resources/

Essa pasta guarda a parte visual e os recursos de interface. É muito importante para explicar Blade, assets e internacionalização.

📌 Pastas e arquivos comuns

  • views/*.blade.php: views Blade renderizadas pelo Laravel.
  • views/layouts/*.blade.php: layouts base reaproveitáveis.
  • views/components/*.blade.php: componentes Blade reutilizáveis.
  • js/*.js ou js/app.js: entrada JavaScript da aplicação.
  • css/*.css ou css/app.css: entrada CSS da aplicação.
  • lang/*.php ou lang/*.json: arquivos de idioma e tradução.

🎓 Leitura de treinamento

Se o aluno quer entender como o Laravel monta a interface, essa pasta vira leitura obrigatória. É aqui que Blade, assets e textos de UI começam a fazer sentido.

🧭 routes/

Essa pasta define os caminhos da aplicação. Ela liga URL, método HTTP e ação que será executada no framework.

📌 Arquivos principais

  • web.php: rotas web com sessão, cookies e renderização de páginas.
  • api.php: rotas de API, normalmente stateless.
  • console.php: definição de comandos Artisan via rota de console.
  • channels.php: canais de broadcast e autorização de eventos em tempo real.

🎓 Leitura de treinamento

Rotas são um ponto-chave para o aluno visualizar o fluxo da aplicação: quem recebe a requisição, para onde ela vai e qual resposta retorna.

🧱 storage/

Essa pasta armazena dados gerados durante a execução da aplicação. Ela sustenta logs, cache, sessões e arquivos produzidos em runtime.

📌 Pastas comuns

  • logs/laravel.log: log principal da aplicação.
  • framework/cache/*: cache interno do framework.
  • framework/sessions/*: arquivos de sessão quando esse driver é usado.
  • framework/views/*: views compiladas do Blade.
  • app/public/*: arquivos enviados e publicados para acesso público.

🎓 Leitura de treinamento

É importante mostrar ao aluno que muito do que o Laravel gera em execução vai para storage/, e que essa pasta não é o lugar principal para editar código.

🧪 tests/

Essa pasta centraliza os testes automatizados. É ela que ajuda o projeto a evoluir com mais segurança e previsibilidade.

📌 Estrutura e arquivos comuns

  • Feature/*.php: testes de fluxo funcional da aplicação.
  • Unit/*.php: testes unitários mais isolados.
  • TestCase.php: classe base para os testes do projeto.
  • Pest.php: configuração base quando o projeto usa Pest.

🎓 Leitura de treinamento

No treinamento, essa pasta mostra que Laravel não serve só para criar telas: ele também sustenta qualidade e manutenção por meio de testes.

📦 vendor/

Essa pasta guarda o framework Laravel e todas as dependências instaladas pelo Composer. Ela é gerada automaticamente e normalmente não deve ser editada manualmente.

📌 O que normalmente existe aqui

  • laravel/framework/*: código interno do framework Laravel.
  • composer/*: arquivos de autoload e metadados do Composer.
  • vendor/autoload.php: autoload das classes do projeto e dependências.
  • pacotes-de-terceiros/*: bibliotecas instaladas no projeto.

🎓 Leitura de treinamento

O aluno precisa saber que essa pasta existe e é importante, mas também precisa aprender cedo que o lugar de trabalho dele normalmente não é vendor/, e sim o código da própria aplicação.

🛠️ artisan

É o executável principal da linha de comando do Laravel. Sempre que o aluno roda um comando Artisan, ele parte desse arquivo.

📌 Para que serve

  • php artisan serve: sobe o servidor local.
  • php artisan migrate: aplica migrations.
  • php artisan make:controller NomeController: cria controllers.

🔐 .env

Arquivo de configuração local com dados sensíveis e variáveis de ambiente. É um dos arquivos mais importantes do projeto.

📌 O que costuma aparecer

  • APP_NAME, APP_ENV, APP_KEY
  • DB_CONNECTION, DB_HOST, DB_DATABASE
  • MAIL_MAILER e variáveis de e-mail

🧪 .env.example

Modelo base de ambiente usado para onboarding do projeto. Normalmente é copiado para gerar o .env.

📌 Papel no time

Ajuda novos desenvolvedores a saber quais variáveis precisam existir sem expor segredos reais do ambiente.

📦 composer.json

Arquivo que define dependências PHP, autoload e scripts do Composer. Ele é a base do gerenciamento de pacotes do projeto Laravel.

📌 Itens comuns

  • require: dependências de produção.
  • require-dev: dependências de desenvolvimento.
  • autoload: namespaces e carregamento automático.
  • scripts: ações automáticas do Composer.

🧷 composer.lock

Arquivo que trava as versões exatas das dependências instaladas pelo Composer para manter consistência entre ambientes.

📌 Leitura de treinamento

O aluno precisa entender que esse arquivo não é para edição manual. Ele existe para garantir repetibilidade na instalação do projeto.

📁 package.json

Arquivo de dependências Node.js do projeto. Ele fica importante quando o Laravel usa Vite, Alpine, Inertia ou outro frontend moderno.

📌 O que normalmente aparece

  • devDependencies: bibliotecas de frontend e build.
  • scripts: comandos como dev e build.
  • laravel-vite-plugin, vite, frameworks JS quando usados.

⚡ vite.config.js

Arquivo de configuração do Vite. Ele define como os assets de frontend serão compilados e servidos durante o desenvolvimento.

📌 Papel técnico

Conecta entradas como resources/css/app.css e resources/js/app.js ao processo de build do projeto.

🧪 phpunit.xml

⚙️ Laravel 11

Lançado em 12 de março de 2024, com suporte a PHP 8.2 até 8.4. Trouxe uma aplicação nova mais enxuta e uma experiência mais organizada para iniciar projetos.

✨ O que mudou

  • 🧱 Estrutura simplificada com menos arquivos padrão na aplicação nova.
  • ❤️ Health routing para checagem rápida de saúde do sistema.
  • 📡 Chegada do Laravel Reverb para WebSockets em tempo real.
  • ⏱️ Rate limiting por segundo e melhorias em filas e criptografia.

🎓 Leitura para aula

  • 📌 Excelente para explicar por que a estrutura do Laravel moderno mudou.
  • 🚀 Mostra um bootstrap mais limpo e focado em produtividade.
  • 🔧 Ajuda a justificar por que alguns arquivos não vêm mais por padrão.

🧩 Laravel 12

Lançado em 24 de fevereiro de 2025, com suporte a PHP 8.2 até 8.5. Foi uma versão de manutenção forte, mas com grande destaque para os starter kits e a experiência de autenticação.

✨ O que mudou

  • 🧰 Novos starter kits para React, Vue, Svelte e Livewire.
  • 🔗 Kits com Inertia 2, TypeScript, Tailwind e componentes prontos.
  • 🔐 Variante com WorkOS AuthKit para SSO, passkeys e login social.
  • 📦 Breeze e Jetstream deixaram de receber evolução como foco principal.

🎓 Leitura para aula

  • 📌 Perfeito para conectar a introdução com o nosso bloco de kits de frontend.
  • 🌉 Ajuda a explicar por que Inertia ganhou tanto peso no ecossistema.
  • 🧭 Mostra uma fase em que Laravel organiza melhor a entrada para stacks modernas.

🤖 Laravel 13

Lançado em 17 de março de 2026, com suporte a PHP 8.3 até 8.5. O foco passou a incluir fluxos AI-native, APIs mais expressivas e recursos modernos de busca e serialização.

✨ O que mudou

  • 🤖 Laravel AI SDK oficial para texto, áudio, imagem, embeddings e agentes.
  • 🧾 JSON:API resources nativos no framework.
  • 🔎 Busca semântica e vetorial integrada ao ecossistema Laravel.
  • 🏷️ Mais suporte a PHP attributes em controllers, filas e autorização.
  • 🛡️ Reforço em proteção contra request forgery e roteamento de filas por classe.

🎓 Leitura para aula

  • 📌 Mostra que Laravel não ficou preso ao CRUD tradicional.
  • 🚀 É uma ótima vitrine para falar de IA sem sair do ecossistema do framework.
  • 🧠 Ajuda a mostrar a transição do Laravel para APIs e fluxos mais inteligentes.

app/Http/Controllers/AdminController.php

<?php

namespace App\Http\Controllers;

use Illuminate\Http\Request;
use App\Models\User; // <--- OBRIGATÓRIO: Importar o Model User

class AdminController extends Controller
{
    // Método responsável pela página de relatórios
    public function relatorios()
    {
        // 1. Buscar totais (contagem)
        $totalUsers = User::count();
        $totalAdmins = User::where('role', 'admin')->count();
        $totalRegular = User::where('role', 'user')->count();

        // 2. Buscar a lista de usuários com paginação (10 por página)
        // Usamos 'paginate' para o {{ $users->links() }} funcionar na View
        $users = User::orderBy('id', 'desc')->paginate(10);

        // 3. Enviar tudo para a View 'admin.reports'
        return view('admin.reports', compact('totalUsers', 'totalAdmins', 'totalRegular', 'users'));
    }
}

app/Http/Controllers/AuthController.php

<?php

namespace App\Http\Controllers;

use App\Models\User;
use Illuminate\Http\Request;
use Illuminate\Support\Facades\Auth;
use Illuminate\Support\Facades\Hash;

class AuthController extends Controller
{
    // --- REGISTRO ---
    public function showRegister() {
        return view('auth.register');
    }

    public function register(Request $request) {
        $dados = $request->validate([
            'name' => 'required',
            'email' => 'required|email|unique:users',
            'telefone' => 'required',
            'password' => 'required|min:4'
        ]);

        $user = User::create([
            'name' => $dados['name'],
            'email' => $dados['email'],
            'telefone' => $dados['telefone'],
            'password' => Hash::make($dados['password'])
        ]);

        Auth::login($user); // Loga automaticamente
        return redirect()->route('dashboard');
    }

    // --- LOGIN ---
    public function showLogin() {
        return view('auth.login');
    }

    public function login(Request $request) {
        $credentials = $request->validate([
            'email' => ['required', 'email'],
            'password' => ['required'],
        ]);

        if (Auth::attempt($credentials)) {
            $request->session()->regenerate();
            return redirect()->route('dashboard');
        }

        return back()->withErrors([
            'email' => 'Credenciais inválidas.',
        ]);
    }

    // --- LOGOUT ---
    public function logout(Request $request) {
        Auth::logout();
        $request->session()->invalidate();
        $request->session()->regenerateToken();
        return redirect()->route('login');
    }
}

app/Http/Middleware/IsAdmin.php

<?php

namespace App\Http\Middleware;

use Closure;
use Illuminate\Http\Request;
use Symfony\Component\HttpFoundation\Response;

class IsAdmin
{
    /**
     * Handle an incoming request.
     *
     * @param  \Closure(\Illuminate\Http\Request): (\Symfony\Component\HttpFoundation\Response)  $next
     */
    
    public function handle(Request $request, Closure $next): Response
    {
        // Se não estiver logado OU não for admin...
        if (!auth()->check() || auth()->user()->role !== 'admin') {
            abort(403, 'Acesso negado: Apenas administradores.');
        }

        return $next($request);
    }
}

<?php

namespace App\Models;

// use Illuminate\Contracts\Auth\MustVerifyEmail;
use Illuminate\Database\Eloquent\Factories\HasFactory;
use Illuminate\Foundation\Auth\User as Authenticatable;
use Illuminate\Notifications\Notifiable;

class User extends Authenticatable
{
    /** @use HasFactory<\Database\Factories\UserFactory> */
    use HasFactory, Notifiable;

    /**
     * The attributes that are mass assignable.
     *
     * @var list<string>
     */
    protected $fillable = [
        'name',
        'email',
        'password',
        'telefone', // Adicionar aqui
        'role'
    ];

    /**
     * The attributes that should be hidden for serialization.
     *
     * @var list<string>
     */
    protected $hidden = [
        'password',
        'remember_token',
    ];

    /**
     * Get the attributes that should be cast.
     *
     * @return array<string, string>
     */
    protected function casts(): array
    {
        return [
            'email_verified_at' => 'datetime',
            'password' => 'hashed',
        ];
    }
}
